§1 · 系統使用者角色（3 種人）

下表 = 系統使用者角色。Edge / Gateway / Skill 等內部系統元件請看下方架構圖。

Role Goal Actions
Expert 醫師 / 知識工作者
  • Input know-how / knowledge（餵自己的腦給 Edge）
  • Scale service（一個分身服務多個 end user）
  • 填 Persona / 餵 Personal-wiki P1
  • 簽核 Edge 草擬的文章 P1
  • LINE chat 跟 Edge 對話、下指示 P2
End User 顧客 / 讀者 / 病人
  • On-time service
  • Personalized service
  • Read articles（LINE OA / FB / WordPress）P1
  • Chat LINE bot 跟 Edge 對話 P2
Admin platform ops（Iven / 未來公司）
  • Maintenance（平台維運、deploy）
  • Track usage（用量監控 / 計費 / 健康度）
  • Push deploy（Zeabur / CF Workers）
  • 開新 Edge 帳號（P1 手動，P2 自助?）
  • 監控 Gateway / Skill / Edge runtime 健康度

DB（Supabase）
PostgreSQL + Auth · 一庫多租戶
多租戶隔離方式
schema-level multi-tenant（不是物理分庫）
每筆資料用 client_id 隔離；RLS 政策確保 Edge 只看自己的資料。

主要表：
  • clients（= Edges）· personas · topics · articles
  • documents（含 client_id IS NULL = 全域共用平台級資產）
  • associations / association_clients（未來醫師聯合的伏筆）
⚠️ Edge Console 直連 Supabase 只用於 Notion sync + SSR；RLS 隔離資料一律走 Gateway
Expert flow（input · train 1:1）
End User flow（serve 1:N）
Shared system（多 Edge 共用，by client_id 隔離）
Internal system（Edge 是核心 AI agent，不是 role）
四句話記住：
① 系統使用者 3 種角色：Expert · End User · Admin
② Edge = 每位 expert 一個專屬 AI agent，吸收 expert know-how 後對外 scale 服務 end user（train 1:1，serve 1:N）
③ Phase 1：expert 走 Console 管 Edge、end user 走「單向收文章」；Phase 2：雙方都走 LINE chat 跟 Edge 對話
④ Gateway / Skills / DB 是 多 Edge shared 基礎設施（schema-level multi-tenant，by client_id 隔離）
