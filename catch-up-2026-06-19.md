## 2. Iven 6/19 commits（LINE Channel Credential security 第一波答案）

引用 `ADR-line-multitenant-decouple-v0.2` + `SM-001`（兩份 spec Iven 本地有，**還沒 push**）。

| Task | Repo | Commit | 內容 |
|---|---|---|---|
| **T1** | gateway | `6cf69e0` | `lineResolver.ts`：webhook destination → 查 `clients.settings.channels.line` → 解 `{tenant_id, channel_secret, channel_access_token, oa_name}`，Cache TTL 300s，回 discriminated union（ok/not_found/disabled/duplicate/missing_secret） |
| **T1** | gateway | `caf924c` | `edgeAuthClient.ts`：EDGE_AUTH_URL / EDGE_AUTH_SHARED_SECRET 移 Gateway 鑄租戶 JWT 用 |
| **T1** | gateway | `f08884c` | `lineWebhook.ts`：resolver 命中 → 用該 OA secret 驗 X-Line-Signature → 鑄租戶 JWT → forward 注入 `Authorization: Bearer` + `X-Qoome-{Tenant-Id, Oa-Name, Line-Destination}`；resolver miss = legacy env secret fallback（**dr-lin 零中斷**）；簽章無效一律 200 drop + log（不回 401） |
| **T2** | gateway | `1803e8f` | `/api/v1/line/{reply, push, loading}` 三個 reply API（ADR-LINE-002 Option C） |
| **T4** | gateway | `98d3b5c` | `/api/v1/intake/*` body.tenant_id 必須匹配驗過的 JWT tenant，不符 403；proxy 加 bodyOverride 避免重讀 stream |
| **T6** | gateway | (同 98d3b5c) | GET `/config` 對 `channels.line.{channel_secret, channel_access_token}` + `channels.wordpress.auth_header` redact 成 `__REDACTED__`；PATCH 收到 sentinel 還原現值（避免整包 round-trip 洗掉 secret） |
| **T5** | campaign | `2e40853` | `jwt_verify.py` JWKS ES256 驗 Bearer → `effective_tenant` 從 token 取、body.tenant_id 不符 403、`INTAKE_ENFORCE_JWT` 預設 true（false 可暫退 Phase B cutover 安全閥） |

**新引入元件**：
- `lineResolver`（Gateway 內）
- `edgeAuthClient`（Gateway → Edge Auth seam）
- `jwt_verify`（Skill 端，JWKS ES256）

**新 Gateway 路由**：`/api/v1/intake/*`、`/api/v1/line/{reply, push, loading}`

## 3. v1 架構誤解 / 缺漏 audit（12 條）

### 🔴 大誤解（§2 主圖 vs moc.md 自相矛盾）

**#1 Edge = Hermes-based Runtime（過早 commit Hermes）**
- §2 寫「`qoome-edge-agent` · Hermes-based · 部署 CF Worker / VPS (TBD)」
- moc.md 自己寫「**不上 Hermes Agentic Agent**（Phase 2 才接），Phase 1 先做保底版（Supabase JWT + Gateway 呼 Skill）」
- Iven 6/10 也確認：「現階段 Expert LINE OA 主要串 Intake chatbot Agent，之後 Hermes 才串醫生自己的 Line」
- **應改**：Phase 1 Edge = **Edge Console (Next.js) + Gateway 呼 Skill**；Phase 2 才升 Hermes Agentic

**#2 Edge 職責 = intent recognition + skill selection（過早 commit）**
- §2 寫「Edge 職責（信條 #3）= 腦/策略 = intent recognition + skill selection，呼 Gateway 執行」
- Phase 1「腦」其實在 **Intake skill (LangGraph)** + n8n channel core；Edge runtime 的「腦」要等 Phase 2 Hermes
- **應改**：Phase 1「腦」在 Skill 內；Phase 2 才回到 Edge runtime

### 🟡 過時 / 缺漏

**#3 林/鄭/凌越 Intake 共用 Dify chatflow（過時）**
- §2 Edge details 寫「林/鄭/凌越 Intake 邏輯共用 **Dify chatflow**」
- MASTER-SYSTEM-MAP §3 元件清冊：**Dify ⚠️ DEPRECATED，遷 LangGraph 中**
- **應改**：共用 **LangGraph chatflow**（`qoome-campaign`），呼叫時注入 tenant identity

**#4 Intake / nuwa skill ⚠️ 缺（過時）**
- §2 Skills box 寫「Intake / nuwa ⚠️ 缺（Iven 後端尚未 expose）」
- 6/19 Gateway 加 `/api/v1/intake/*` + campaign T5 JWT verify → **Intake 已 expose**
- **應改**：Intake ✅ expose；nuwa 仍待（不同東西，nuwa 是 persona/skill extraction 工具）

**#5 Gateway 只有 `/edge/*` + `/studio/*` 兩個 namespace（缺）**
- 6/19 新增 `/api/v1/intake/*` + `/api/v1/line/{reply, push, loading}`
- **應改**：Gateway namespace 補 2 條

**#6 缺 Edge Auth shared box（缺）**
- §2 shared chain 只有 Gateway → Skills → DB
- MASTER-SYSTEM-MAP §3 第 2 元件 = **qoome-edge-auth**（CF Worker，Token Issuer：tenant_id → Supabase ES256 JWT）
- **應改**：shared chain 加 **Edge Auth** 第 4 box（在 Gateway 旁邊或 DB 上方）

**#7 Knowledge 分類只有 2-3 層（缺 Methodology 獨立類）**
- §2 Edge details 寫「shared_libraries（醫療通用知識庫）/ private_library_id（私有）/ Personal-wiki」
- Iven 6/10 給 **4 分類**：Librarian-Global / Librarian-Private / **Methodology**（獨立類）/ Personal-wiki
- **應改**：Knowledge 分層補 4 分類，特別標 Methodology 是獨立一類

**#8 Channels settings 沒提加密邊界（缺）**
- §2 Edge details 寫「channels settings：LINE OA / FB / WordPress 各自帳號 credentials」
- 6/19 T6 給明確契約：**值放 Gateway DB，Console 只 UI；GET 自動 redact，PATCH 用 sentinel 還原**
- **應改**：加註腳「實際 secret 存 Supabase + redact/sentinel pattern (T6)；Edge Console 不暴露明文」

**#9 Late.io 完全沒畫（缺）**
- §1 End User actions 寫「Read articles (LINE OA / FB / WordPress)」、§2 也只列「channels settings」
- Iven 6/10：「**FB 等 Social Media 走 Late.io 統一閘道**」P1 已定
- **應改**：加 Late.io adapter（FB/IG/social 出口）；WordPress 直發；LINE OA 直發

**#10 LINE ingress chain 沒畫（缺）**
- §2 沒呈現 End User → LINE OA → Gateway lineWebhook → resolver → JWT → Skill 這條 ingress chain
- **這條 Phase 1 dr-lin Intake 對話走的就是這條**，不只 Phase 2
- **應改**：加 ingress flow callout（含 6/19 T1 新元件）
