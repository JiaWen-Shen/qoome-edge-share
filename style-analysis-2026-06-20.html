§1. 框架：顯性 vs 隱性風格

EMNLP 2025 Findings paper "Catch Me If You Can? Not Yet: LLMs Still Struggle to Imitate the Implicit Writing Styles of Everyday Authors"（arxiv 2509.14543）量化過：LLM 用 prompt-only / few-shot in-context 模仿真人作者時，顯性層命中 ~60-70%，隱性層卡在 30-40%，且加範例、換大 model 都無法突破天花板。

顯性風格（surface）隱性風格（implicit）
作者「能說得出口」✅ 5 步框架、固定比喻、catchphrase、禁用詞、修辭手法❌ 思考節奏、何時停、何時略、tone 何時嚴肅
Prompt 工具可傳遞✅❌
Paper 命中率~60-70%~30-40%（ceiling）
突破方法—Adapter (LoRA) 或 Activation Steering（Persona Vectors）
關鍵理論結論：prompt 模仿風格有結構性天花板——再細的 system prompt 也只能改善顯性層，碰不到隱性層。要破天花板，要從「指令 model 怎麼做」改成「改變 model 的本能」。

三個收斂方向

對「語料稀少」

業界主流走兩條：

  • Adapter mixing（AuthorMix）—— 從 high-resource adapter mix 出風格，極少樣本即可
  • Few-shot in-context（2-5 例啟動）—— 顯性可、隱性難（受 §1 ceiling 限制）

更可靠：adapter 路線。

對「死板」

2025 主流：activation-space steering（Persona Vectors）

  • 把人格編碼成向量、推論時動態注入
  • 比靜態 prompt 自然 + 長 context 不衰減

prompt 路線到頂、要往 model 內部走。

#論述適用場景
1 質化任務需要不同的工具鏈——內容創作、品牌語氣、persona 模擬這類任務跟工程任務不同，沒有單元測試這種客觀對錯標準。LLM 無法自我驗證結果是否符合期待，容易中途收尾。因此需要建立明確的品質標準（rubriciRubric：明文化的評分標準，原為教育領域評量量表；在 AI 工程裡指「拆成多維度、每維度有明確 reference 的品質標準」，可由人或 evaluator agent 對照使用。）作為外部對照。 內容創作 / persona 模擬 / 品牌語氣場景
2 Prompt 工程本身有結構性上限——最新學術研究（EMNLP 2025）量化證明，prompt-only 方法模仿「隱性風格」（思考節奏、tone 起伏、何時用比喻）只能達到 30-40%；加更多範例、換更大模型都無法突破。投入更多時間在 prompt engineering 收益遞減。 判斷「再 tune 一次 prompt」是否值得時的依據
3 evaluation-driven workflow 是 2025-2026 業界共識方向——從業界實作（Gary 連續 27 小時實驗）、機構研究（Anthropic 內部美術館評測）、到學術成果（Persona Vectors、PersonaCite、TwinVoice），都收斂到「拆 rubric → evaluator agentiEvaluator Agent：專責評分的 AI 角色，獨立於生成端，依 rubric 對輸出打分、判斷通過或退回，類似 code review 機器人。 → 迭代」這條路徑。 論證方法論的主流性與穩定性
4 方法論的投入有複利效應——rubric 文件本身是組織內部品質標準的知識資產；累積的失敗案例 corpusiFrown Corpus：英文 frown = 皺眉；指「評估時皺眉的範例集合」——保留失敗案例與不通過的原因，作為 rubric 演化的料與未來模型訓練的反例資料集。（失敗案例 + 不通過原因）將來能成為模型訓練素材。當期投入的評估工作，未來會持續產生 leverage。 說明投入價值的時間軸與複利效益
5 跟 AI 自動化的關係——evaluator gate 不是限制 AI 自動化，是讓自動化能持續產出可用結果。沒有評估機制的 AI workflow 在多篇後品質容易崩塌；有評估機制的能維持穩定（Gary 27 小時連續產出實驗已驗證）。 說明品質機制與自動化目標並不衝突

不同 topic 卻 verbatim 套用同一比喻——比喻清單被當 must-use checklist。

2. Connective 詞反覆使用

說穿了、坦白講、更粗的是、問題是 兩篇都密集使用。LLM 學到 surface lexicon，但缺「何時不用」的判斷力。

3. 5 步框架機械執行

不同 topic 但段落順序與轉折節奏一致——framework 被當 template fill-in，不是「思考順序」。

場景開頭（凌晨一點/凌晨兩點） → 你以為 X 是 Y,其實不是 → 機制解釋 + 比喻 → 研究引用（PMID 格式） → 「我在診間看過」鋪陳 → 個人化建議 → 末段「今晚試試」+ 一句「你的身體本來就...」收尾

4. 錯字 / 邏輯錯誤透露 publish flow 缺 review gate

  • 他要不要吞兩顆呢?（應是「我」）
  • 更粗的是（應是「糟」，反覆出現）
  • 在固定時候訊號（應是「時間」）

真人醫師絕不會放過這種錯——顯示 AI 直接 publish，無 review gate。

5. Tone 沒有起伏

整篇同一「溫暖權威」tone。真人講副作用 / 警告會自動嚴肅，講個人經驗會放鬆——AI 整篇平直。

§5. 提案：兩階段執行路線

Phase 1 — 優化的生成機制（先治當期問題）

目標

Phase 2 — variation × evaluation（Karen + Iven 共同執行）

核心構想

同一主題用優化過的 persona 生成多個 variation，由 Karen 與 Iven 共同評分，蒐集判斷與皺眉點作為下游 rubric 與 persona refinement 的原料。

具體動作

  1. 每主題生成 3-5 個 variation——同一份 persona、同一個主題，控制變數逼出隱性風格差異。variation 變數的具體設計留待 Phase 1 完成後，Karen 視 Phase 1 baseline 表現再規劃。
  2. 做一個內部 evaluation webpage——簡單 web form，呈現主題 + 多個 variation 並列、提供評分欄位（排序、highlight 皺眉句、偏好標記、自由 comment）。Karen 跟 Iven 各自登入評分，資料統一收進後端。
  3. 累積 frown corpus——每筆配對保留：variation 內容 + 雙方評分 + 皺眉原因 + 偏好理由。
  4. 從 corpus 萃 rubric 細節——累積到一定量（建議 ~50 筆）後，把皺眉原因歸類，產出可被 evaluator agent 執行的 rubric 維度與閾值。
  5. rubric 回灌 Phase 1——學到的細節寫回 nuwa v4 的 metaphor_bank.usage_rules / voice_architecture.surface_rules / content_boundaries，產出 v4.1。

Owner

Karen 負責 webpage 開發、variation 設計、corpus 結構化、rubric 萃取；Iven 共同評分作為領域權威 ground truth 來源。
